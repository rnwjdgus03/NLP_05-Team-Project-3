# 멘토 회의 브리핑 — 뉴스 수치 주장 KOSIS 사실검증

- 일자: 2026-07-21 · B팀
- 목적: 현재 방향이 맞는지 확인, 남은 의사결정 자문

---

## 1. 한 줄 요약

조선일보 뉴스의 수치 주장을 KOSIS(국가통계포털) 통계와 자동 대조해 사실검증하는 파이프라인을 **코드 기반으로 재구축**했고, 파일럿까지 완주했다. 지금은 **정확도를 측정할 골드셋 라벨링**이 크리티컬 패스이며, 그 결과로 세부 설계(검색 방식·게이트 임계값)를 확정할 단계다.

## 2. 문제 정의 & 접근

- 초기 문제: A팀이 뽑은 데이터에 오류·빈칸이 많고 KOSIS 표 매칭이 실패. 원인은 조악한 컬럼 설계와 KOSIS 좌표체계(표×축×항목×시점) 미이해.
- 전환: 규칙기반 대신 **HCX(HyperCLOVA X)로 구조화 추출 + 재현 가능한 코드 파이프라인**. 측정값 단위(값 하나당 한 행)로 스키마 재설계.
- 학술 프레임: FEVER(3단계 사실검증), Scrutinizer(통계 claim 검증), BLINK(검색+재순위) 계열로 정렬.

## 3. 파이프라인 (7단계)

```
① 전처리      뉴스 본문 → 문장 분리 + 문맥 부착
② is_claim    KOSIS로 검증 가능한 수치 주장만 True (HCX-007)
③ 추출        문장을 "값 하나당 한 행(measurement)"으로 구조화
④ 게이트      실제 검증 대상만 통과 (코드 규칙: KOSIS값·국내·시점완비)
⑤ 검색        107,138개 표에서 후보 탐색 (lexical / hybrid BGE-M3)
⑥ meta 해소   후보 표의 축·항목 코드 확정 (KOSIS getMeta)
⑦ 검증        KOSIS 실제값 조회 → 주장값과 대조 → 일치/불일치/판단불가
```

## 4. 파일럿 결과 (실측)

원천 **조선일보 기사 2,705건 → 정제 문장 47,514건**. 파일럿 표본으로 두 실행:

| | lexical 실행 | hybrid 실행 (codex) |
|---|---|---|
| 표본 | 51 claim | 100 measurement (47 claim) |
| 게이트 통과(ready) | 33 | 22 |
| 검색 방식 | 어휘 | BGE-M3 임베딩+reranker |
| 검증 결과 | 일치 4 · 불일치 5 · 판단불가 24 | 자동판정 0 · 검토대기 15 · 반려 7 |

- 검증까지 **완주**했고, KOSIS API(목록·메타·자료 3종) 실호출로 실제값 대조 성공.
- pytest 80개 통과, 파이프라인 재현성 확인.

## 5. 핵심 발견 3가지

1. **게이트가 대부분을 걸러내는 건 정상이다.** 문장 대부분은 KOSIS로 검증 불가한 값(정책값·해외·기업·전망·파생지표). 파일럿에서 measurement 114개 중 33개만 통과. 이는 실패가 아니라 "검증 가능한 것만 남기는" 설계 의도.

2. **판단불가의 최대 원인은 KOSIS 커버리지 한계다.** 반도체·자동차·바이오헬스 같은 MTI 주력품목 수출액은 **KOSIS에 표 자체가 없음**(SITC·KSIC만 존재, MTI는 관세청 소관). 코드로 해결 불가 — 커버리지 밖으로 분류하는 게 정답. 반면 "LCC 10사 합계" 같은 묶음은 개별 항공사 합산으로 복구 가능(코드셋 구축 완료).

3. **검색 방식의 트레이드오프.** lexical은 단어 겹침만 봐 엉뚱한 표를 1위로 올리기도 함. hybrid는 의미 검색이라 표를 잘 찾지만 보수적이라 자동판정 대신 사람 검토(REVIEW)로 라우팅. → hybrid 우세하나 확정은 골드 채점 후.

## 6. 현재 의사결정

- **검색은 hybrid로 기울되 lexical은 폴백으로 유지** (hybrid는 임베딩 인덱스 필요, 인덱스 없을 때 대비).
- **골드셋으로 정확도 측정 후 확정**: 게이트 임계값·자동판정 공격성·검색방식을 숫자로 결정.
- 골드는 검색방식과 무관하게 **한 세트면 두 방식 모두 채점** 가능. 입력을 통일해 라벨 비용을 한 번으로.

## 7. 남은 작업 & 로드맵

| 우선순위 | 작업 | 상태 |
|---|---|---|
| 1 | 골드셋 라벨링 (정답지 = 정확도 측정) | 진행 중 (팀) |
| 2 | 골드 채점 → 검색방식·게이트 확정 | 골드 대기 |
| 3 | 코드북을 hybrid에 override로 통합 (확정 매핑 즉시조회) | 설계됨 |
| 4 | 파일럿 검증 후 전량(47,514) 확장 | 골드 통과 시 |

## 8. 멘토님께 확인받을 질문

1. **게이트·판단불가 비율**: measurement의 70%가 걸러지고 판단불가가 많은데, 이 보수적 설계 방향이 맞나? "검증 가능한 것만 정확히"가 옳은 전략인지.
2. **KOSIS 커버리지 밖 처리**: 반도체 MTI처럼 KOSIS에 없는 값은 관세청 등 출처 확장 vs 범위 제한(KOSIS 한정) 중 어느 쪽이 프로젝트 목표에 맞나.
3. **검색 방식**: hybrid(임베딩) 채택이 타당한지, 자동판정 임계값을 얼마나 공격적으로 둘지.
4. **평가 규율**: 골드셋 동결·train(코드북)/test(골드) 분리·recall@k/정밀도 지표 구성이 적절한지. (지난 힌트였던 Structured Outputs는 반영 완료.)
