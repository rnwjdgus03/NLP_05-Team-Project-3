# 골드 라벨 스펙 — 새 measurement 파이프라인 정확도 측정

- 작성: B팀, 2026-07-21
- 목적: codex가 올린 파이프라인 성능을 **단계별로** 잴 수 있게 골드 라벨의 대상·컬럼·측정법을 고정한다.
- 원칙: "채워졌나"가 아니라 "맞게 채워졌나"를 잰다. 각 단계는 독립적으로 채점한다(앞 단계 오류가 뒤로 전파되지 않게, 정답 기준으로 끊어서 측정).

골드는 **두 시트**면 전부 커버된다.
- **시트 A (claim 단위)** = is_claim 골드
- **시트 B (measurement 단위)** = 추출 게이트 + 검색 + 판정 골드 (셋 다 measurement 단위라 한 시트)

`make_gold_templates.py`가 파이프라인 산출물에서 이 두 시트를 모델 예측값이 채워진 채로 뽑아준다. 사람은 `gold_` 컬럼만 채운다.

---

## 시트 A — is_claim 골드 (①단계)

**대상 CSV**: `is_claim_005vs007_v12_diff28.csv` (005·007이 갈린 28건, 이미 `gold_label` 빈 칸 존재)
- 추가로 **랜덤 100건**을 라벨해야 "절대 정확도"가 나온다. diff28은 두 모델 승패만 알려줄 뿐, 전체 정확도가 아니다.

**채울 컬럼**
- `gold_label` ∈ {`True`, `False`} — 이 문장이 KOSIS 통계로 검증 가능한 수치/추세 주장인가.
  - 판단 기준은 `is_claim_filter_hcx.py`의 SYSTEM_PROMPT True/False 조건과 동일하게 적용(해외·기업·여론조사·전망·정책목표 = False).

**측정법**
- 절대 정확도 = `mean(gold_label == is_claim_007)` (랜덤 100건 시트에서).
- 모델 비교(28건 diff): 005가 맞은 수 = `sum(gold_label == is_claim_005)`, 007이 맞은 수 = `sum(gold_label == is_claim_007)`. 큰 쪽이 우세.
- True 클래스 precision/recall도 함께 보고(검증 대상 누락률 파악).

## 시트 B — measurement 골드 (②③④단계)

**키**: `claim_measurement_id` (measurement 행 단위, 값 하나당 한 행)
**대상**: `hcx_v15.csv`의 모든 measurement 행. 검색·판정 골드는 이 중 **게이트 통과(ready)** 행에만 채우면 된다.

### ② 추출 게이트 정밀도

**채울 컬럼**
- `gold_verifiable` ∈ {`Y`, `N`} — 이 measurement가 **KOSIS 시계열로 실제 검증 가능한 값**인가.
  - `N` 예: 정책 목표·고시값(최저임금), 조건값(주 40시간), 해외·기업 통계, 순위, 단순 횟수.
- `gold_measurement_correct` ∈ {`Y`, `N`} — 추출된 `measurement_indicator / value / measurement_period / unit / measurement_role`이 문장과 **모두** 맞는가. 하나라도 틀리면 N.

**측정법** (게이트 통과 = `mapping_eligible == Y`, 즉 ready에 들어간 행)
- **게이트 정밀도(precision)** = `P(gold_verifiable=Y | 게이트 통과)` = 통과시킨 것 중 진짜 검증가능 비율 → 거짓양성(엉뚱한 값을 검증 대상으로 넘김) 방지.
- **게이트 재현율(recall)** = `P(게이트 통과 | gold_verifiable=Y)` = 진짜 검증가능값 중 통과된 비율 → 거짓음성(놓친 검증가능값) 파악.
- **추출 필드 정확도** = `mean(gold_measurement_correct=Y)` (ready 행 대상). 값·시점·지표가 실제로 맞는 비율.
- 반려 사유 분포(`mapping_exclusion_code`)를 골드와 대조: `gold_verifiable=Y`인데 반려된 행 = 게이트가 너무 빡빡한 지점(예: `BINDING_NOT_CONFIRMED`, `PERIOD_MISSING`).

### ③ 검색 recall@k

**채울 컬럼** (ready 행에만)
- `gold_org_id`, `gold_tbl_id` — 이 값이 실제로 들어있는 정답 KOSIS 표.
- `gold_obj_l1`, `gold_itm_id` — 정답 분류축 코드·항목 코드(메타 매칭용). 모르면 비워도 됨(표 단위 recall만 측정).

**측정법** (`*_table_candidates.csv`의 `candidate_rank`와 조인)
- **recall@k** = `(정답 tbl_id가 후보 top-k 안에 있는 measurement 수) / (gold_tbl_id 채워진 measurement 수)`. k=1,5,10 보고.
  - 계산: measurement별로 `tbl_id == gold_tbl_id`인 최소 `candidate_rank`를 찾아, 그 rank ≤ k면 hit.
- **메타 정확도** = `mean(selected_itm_id == gold_itm_id)`, `mean(selected_obj_l1 == gold_obj_l1)` (정답 코드 채운 행 대상).
- recall@k가 낮으면 → 임베딩 인덱스(BGE-M3)가 해결 지점. lexical만으로 recall@5를 먼저 재고, 임베딩 붙인 뒤 재측정해 개선폭을 본다.

### ④ 최종 판정 정확도

**채울 컬럼** (ready 행에만, 사람이 KOSIS를 직접 조회해서)
- `gold_verdict` ∈ {`일치`, `불일치`, `판단불가`}.
- `gold_actual_value` — 사람이 KOSIS에서 찾은 실제 값(참고·재현용).

**측정법** (`*_verified.csv`의 `verdict`와 대조)
- **verdict 정확도** = `mean(verdict == gold_verdict)`.
- **판정 가능 케이스 정확도** = `gold_verdict ∈ {일치,불일치}`인 행만 대상으로 한 정확도(판단불가 제외). 실제로 맞다/틀리다를 얼마나 맞히나.
- 혼동행렬로 오류 유형 확인: 일치를 불일치로(허용오차 문제), 검증가능을 판단불가로(매핑 실패) 등.

## 씨앗(재활용) — 라벨 비용 절감

정현님이 손으로 매핑·검증한 기존 파일이 있으면 ③④ 골드의 상당수를 그대로 재활용할 수 있다.
- `hcx_mapped.csv` (org_id/tbl_id/obj_l1/itm_id/match_status) → `gold_tbl_id` 등 검색 골드 씨앗.
- `outputs/bteam_review/final_verified_enriched.csv` (verdict 포함) → `gold_verdict` 씨앗.
- 단, **재활용 전 사람이 한 번 눈으로 확인**하고 채택한다(과거 코드북 버전 드리프트 이력 있음).

## 최소 규모 권장

- ① is_claim: diff28 전부 + 랜덤 100.
- ②③④ measurement: ready 전체(51건 추출 시 대략 20~40 measurement) + gold_verifiable는 **반려된 행도** 라벨해야 recall 계산 가능(전체 measurement).
- 표 단위(recall@k)는 코드 없이 tbl_id만 있으면 되니 가장 싸다. 값 판정(④)이 가장 비싸다(KOSIS 직접 조회).
